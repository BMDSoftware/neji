becas API

The becas API is the programming interface to becas. You can use the API to integrate your app or website with becas. This allows you to programmatically annotate text or PubMed abstracts with biomedical concepts on the fly.

Concepts Background

The becas API annotates text and PubMed abstracts with biomedical concepts. Each concept belongs to a given semantic group, or entity type.

Every concept has a unique concept identifier, a string like NCBI:9606:T001:SPEC. This unique identifier, or cid, is made up by four parts separated by colons. The first part identifies the data source where the concept is registered; the second part is the concept identifier in that data source; the third part is the sub-group of the concept; and the last part is the concept type.

becas recognizes concepts mentioned in text by various synonyms and orthographic variations. Each recognized instance of a concept in text is an entity. Given the inherent ambiguity of text and existence of various concepts with the same names, some entities can be tagged with multiple concepts, sometimes from different concept groups.

Identified concepts have external references to concept databases. An external reference looks like NCBI:9606. The first part identifies the database and the second part is the concept identifier in that database.

Regardless of the name a given concept is identified by (the entity), every concept has a preferred name, the primary name by which it is known.

The becas API tags entities as being certain concepts and provides information about their preferred names and external references.

Interface

becas exposes annotation services through an HTTP REST API. All services accept input parameters in JSON and return results in JSON, XML, A1 or CoNLL, depending on your needs.

The base URL for all API endpoints is {{baseUrl}}/api/ and the only allowed method is POST.

All requests expect a content type header of Content-Type: application/json.

The API supports compression to reduce transfer times and bandwidth usage. If your client is able to decode gzipped content, send a Accept-Encoding: gzip header with your requests and you'll receive compressed JSON responses.

API Clients New!

You can now easily use the becas API as a cross-platform command-line tool or programmatically from Python through the becas-python official API client. Check out the documentation and the source code. It's open source, and contributions are welcome!

If you develop an open source becas client in any other language and want to make it freely available, please tell us about it and we'll publish it here.

Authentication

The API is free to use for non-commercial purposes, but you are required to authenticate yourself and your application using a contact and a tool name.

You should append two GET parameters to all API endpoints: email and tool. For example, the text annotation service endpoint URL should look like {{baseUrl}}/api/text/annotate?email=<EMAIL>&tool=<TOOL>, where <EMAIL> and <TOOL> are your email address and the name of the tool you're integrating becas into. For testing purposes you can use tool name equal to test.

Note We will never send you spam or give away/sell your email address. It will only be used to contact you in case your application exceeds a reasonable usage quota.

Common Parameters

Apart from the authentication parameters, which should be provided as GET parameters in the URL, all other parameters should be provided in a JSON encoded POST payload.

All API calls accept an optional parameter named groups, specifying the types of entities you want to annotate. It is an object where keys are case-sensitive semantic group identifiers and values are boolean. By default and if ommited, entities from all groups are annotated, which is equivalent to passing the following groups object: 

{
	"groups": {
		"SPEC": true,
		"ANAT": true,
		"DISO": true,
		"PATH": true,
		"CHED": true,
		"ENZY": true,
		"MRNA": true,
		"PRGE": true,
		"COMP": true,
		"FUNC": true,
		"PROC": true
	}
}

Note If you specify a groups object, any group ommited will be assumed to be false. If, for example, you were only interested in tagging disorders and genes/proteins, you could pass the following groups object: 

{
	"groups": {
		"DISO": true,
		"PRGE": true
	}
}

API Calls

The becas API exposes methods to automatically annotate:

Text
Annotate biomedical text fed to the service.
PubMed abstracts
Receive a PMID and return an annotated PubMed article in IeXML.

For each type of content, there are two different types of endpoints:

The annotation endpoints
Return results in JSON only. Provide information about concepts, such as preferred names and external references.
The export endpoints
Return results in JSON, XML, A1 or CoNLL. Tagging of concepts with cids only, no concept metadata. Provide NLP features.

The annotate endpoints are the ones used to power the becas app and widget. The export endpoints provide annotation results in various formats.

All services require authentication and accept a parameters object in a JSON encoded payload. The optional groups parameter, described above is recognized by all services. Parameters specific to each service are described bellow.

Annotating Text

The text annotation and export endpoints perform annotation of biomedical text provided by clients.

The annotation endpoint

{{baseUrl}}/api/text/annotate?email=<EMAIL>&tool=<TOOL>
Features
  • Concept identification
  • Concept metadata:
    • Preferred names
    • External references
  • Maximum depth of nested concepts: 1
    • Deeper nested concepts are resolved to the largest span
Parameters
Name Type Default Description
text string: "..." none (required) Biomedical text to annotate.
echo boolean: true|false false If text should be returned in the response object.
groups object: {"GROUP": true|false, ...} If ommited: all groups true
If some groups present: all others false 		Types of concepts to annotate.
Response

Returns a JSON object with the following keys:

Name Type Description
text string: "..." Text submitted (clean, without annotations), if echo was true.
entities array of strings: ["...", ...] Entities recognized in text, encoded as |-separated, 3-element tuples of
entity|CID1;CID2;...|pos.
where pos is the 0-based position of the first character of entity in text.
ids object of objects: {"CID": {"name": "...", "refs": ["...", ...], ...} Concept metadata, with concept IDs, preferred names and external references.

  • Sentence splitting
  • Maximum depth of nested concepts: 1

XML Response

Returns an IeXML document with a text node nested inside a response root element.
The text element contains a s element for each sentence in the text.
Each s element has entities tagged using e elements and nested/intersected entities are marked using w elements.
Please refer to the IeXML specification for further details.

Response sample: 

<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<response>
  <text>
    <s id="0">
      <e id="UNIPROT:P42771:T116:PRGE">p53</e> biosignatures contain useful information for <e id="UMLS:C0006826:T191:DISO">cancer</e> evaluation.</s>
  </text>
</response>

  • Maximum depth of nested concepts: unlimited

Return Values

Successful service executions return a 200 status code and results in the format you requested.

If something goes wrong you'll receive a JSON response with two fields: status (same as the HTTP response status code) and error, a message describing the problem. For example, if you forget to authenticate your request, you'll receive the following response with a 401 status code: 

{"status": 401, "error": "Please authenticate your request with `email` and `tool` parameters"}

All status codes returned by the API are listed in the following table.

Status code Reason
200 Ok Everything went fine.
400 Client error Your request is malformed, probably missing some parameter.
See the error message for more details.
401 Unauthorized You didn't include both the email and tool authentication parameters.
404 Not found The PubMed abstract you requested does not exist. (PMID not found in MEDLINE)
405 Method not allowed The request method used is not allowed. All services require POST.
413 Request entity too large The request payload is too large. Try again with less text.
429 Too many requests You performed too many requests in a short amount of time.
Please wait the number of seconds specified in the Retry-After header before issuing more requests.
500 Internal server error Something went wrong on our side. Please try again after some time.
503 Service unavailable The service is currently unavailable due to maintenance or server overload. Please try again later.

cURL Samples

cURL is a handy command line tool capable of executing parameterized HTTP requests. If you're using a UNIX based system, like Linux or Mac OS, chances are you already have it installed.

This section contains some cURL snippets you can copy and paste into your terminal to quickly test the becas API. The suggested examples should return some of the response samples illustrated in the API Calls section.

Note The examples bellow use the --compressed flag to make cURL request gzipped responses. If your version of cURL was not compiled with gzip support, simply omit the flag.

All curl samples use the $EMAIL and $TOOL variables in service endpoint URLs. To facilitate testing, please set them before trying the samples. For example: 

EMAIL="you@example.com"
TOOL="curl-samples-test"

Samples

Annotate text with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/text/annotate{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Annotate text with disorders only: 

curl -X POST "{{baseUrl}}/api/text/annotate{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"groups": {"DISO": true}, "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Annotate text with concepts from all supported semantic groups and echo the text back in the response object: 

curl -X POST "{{baseUrl}}/api/text/annotate{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"echo": true, "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Export annotated text in JSON with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/text/export{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"format": "json", "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Export annotated text in XML with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/text/export{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"format": "xml", "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Export annotated text in A1 with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/text/export{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"format": "a1", "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Export annotated text in CoNLL with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/text/export{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"format": "conll", "text": "p53 biosignatures contain useful information for cancer evaluation."}' \
--compressed

Annotate a PubMed abstract in JSON with concepts from all supported semantic groups: 

curl -X POST "{{baseUrl}}/api/pubmed/annotate/23291977{{authSuffix}}" \
--header "Content-Type: application/json" \
--compressed

Annotate a PubMed abstract in JSON with disorders, chemicals and genes/proteins only: 

curl -X POST "{{baseUrl}}/api/pubmed/annotate/23291977{{authSuffix}}" \
--header "Content-Type: application/json" \
--data '{"groups": {"DISO": true, "CHED": true, "PRGE": true}}' \
